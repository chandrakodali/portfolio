GitOps Best Practices with ArgoCD

By Chandra Koushik Kodali ‚Ä¢ 6 min read ‚Ä¢ January 2026

GitOps is a paradigm that has transformed how we manage Kubernetes deployments. Instead of pushing changes to clusters, we declare the desired state in Git and let automated controllers pull and apply those changes. This article covers the best practices I've learned from implementing GitOps at scale.

Whether you're just starting with GitOps or looking to optimize your existing workflows, this guide will help you build a robust, scalable, and auditable deployment pipeline.

What is GitOps?

GitOps is an operational framework that uses Git as the single source of truth for declarative infrastructure and applications. The core principles are:

  1. Declarative: The entire system is described declaratively
  2. Versioned and Immutable: The desired state is stored in Git
  3. Pulled Automatically: Approved changes are automatically applied
  4. Continuously Reconciled: The system self-heals to match the desired state
Core Concept: With GitOps, if you want to know what's running in production, you look at Git. If you want to change production, you change Git.

Push vs Pull: Understanding the Difference

Traditional CI/CD pipelines push changes to clusters. GitOps tools pull the desired state from Git and apply it. This subtle difference has significant implications:

Aspect Push-based (Traditional) Pull-based (GitOps)
Credential access CI has cluster credentials Only the controller has credentials
Drift detection Manual or none Automatic and continuous
Rollback Re-run pipeline or script Revert the Git commit
Audit trail CI logs Git history
Security Cluster credentials in CI Cluster never exposed externally

Zero-Downtime Deployment Strategies

ArgoCD supports multiple sync strategies. Here's how to configure each for zero-downtime:

Rolling Updates (Default)

apiVersion: apps/v1 kind: Deployment spec: strategy: type: RollingUpdate rollingUpdate: maxSurge: 25% # Allow 25% more pods during update maxUnavailable: 0 # Never remove old pods before new ones are ready

Blue-Green with ArgoCD Rollouts

Figure 2: Blue-Green Deployment Flow
apiVersion: argoproj.io/v1alpha1 kind: Rollout metadata: name: my-app spec: strategy: blueGreen: activeService: my-app-active # Service pointing to live version previewService: my-app-preview # Service for testing new version autoPromotionEnabled: false # Require manual promotion scaleDownDelaySeconds: 30 # Keep old pods for 30s after switch

Canary Deployments

Gradually shift traffic from old to new version:

apiVersion: argoproj.io/v1alpha1 kind: Rollout spec: strategy: canary: steps: - setWeight: 10 # Send 10% traffic to new version - pause: {duration: 5m} # Wait and monitor - setWeight: 30 - pause: {duration: 5m} - setWeight: 60 - pause: {duration: 5m} - setWeight: 100 # Full rollout # Automatic rollback on failure analysis: successfulRunHistoryLimit: 3 templates: - templateName: success-rate

Security Best Practices

  1. RBAC for Applications: Use ArgoCD Projects to restrict which repos and clusters each team can deploy to
  2. Sealed Secrets: Never store plain secrets in Git ‚Äî use Sealed Secrets or External Secrets Operator
  3. GPG Signed Commits: Require signed commits for production branches
  4. Branch Protection: Enforce PR reviews for production configurations

Never enable automated sync with prune: true on production without proper safeguards. A misconfigured sync could delete critical resources.

Monitoring GitOps Health

Track these key metrics to ensure your GitOps pipeline is healthy:

  • Sync Status: Number of applications in/out of sync
  • Sync Duration: Time from commit to deployment
  • Drift Occurrences: How often clusters drift from desired state
  • Rollback Frequency: How often you need to revert changes

Key Takeaways

  1. Git is truth: Everything in production should be in Git
  2. Secure your repos: Git repos are now production infrastructure
  3. Use progressive delivery: Canary and blue-green reduce risk
  4. Monitor sync health: Out-of-sync apps indicate problems
  5. Start simple: Begin with a single cluster before scaling
