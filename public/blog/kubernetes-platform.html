Building a Self-Service Kubernetes Platform

In today's fast-paced development environment, engineering teams need the ability to deploy and manage their applications without waiting for operations teams to provision infrastructure. This is where Internal Developer Platforms (IDPs) come in ‚Äî they provide self-service capabilities while maintaining security, compliance, and operational best practices.

In this comprehensive guide, I'll walk you through how to design and build a production-ready Kubernetes platform that has served 200+ engineers across multiple organizations.

What is an Internal Developer Platform?

An Internal Developer Platform (IDP) is a layer on top of your underlying infrastructure that provides developers with self-service access to deploy, manage, and observe their applications. Think of it as an "abstraction layer" that hides the complexity of Kubernetes while exposing only what developers need.

Key Principle: A good platform should feel like a product. Developers are your customers, and their experience determines adoption rates.

Platform Architecture Overview

Before diving into implementation details, let's understand the high-level architecture. Our platform consists of four main layers:

Figure 1: Internal Developer Platform Architecture

Layer 1: Developer Interface

This is what developers interact with daily. It includes:

  • Developer Portal: A web UI (using Backstage or custom) for browsing services and documentation
  • CLI Tool: A custom CLI for common operations like deploying, scaling, and debugging
  • Templates: Pre-configured application templates that follow best practices
  • GitOps Repos: Git repositories where developers define their desired state

Layer 2: Platform Services

This layer provides the automation and tooling that makes self-service possible:

  • ArgoCD: Continuous delivery through GitOps
  • Crossplane: Infrastructure provisioning via Kubernetes APIs
  • Vault: Secrets management and dynamic credentials
  • Cert-Manager: Automatic TLS certificate provisioning

Layer 3: Kubernetes

The orchestration layer that runs your workloads with proper isolation and security:

  • Namespace isolation per team/environment
  • RBAC for access control
  • Network policies for traffic segmentation
  • Service mesh for observability

Layer 4: Infrastructure

Everything managed by Terraform ‚Äî the cloud resources that power your platform.

GitOps Workflow

At the heart of our platform is GitOps. Every change ‚Äî from application deployments to infrastructure updates ‚Äî flows through Git. This provides:

  • Audit trail: Every change is a commit
  • Rollback capability: Just revert the commit
  • Collaboration: Changes go through pull requests
  • Consistency: The repo is the single source of truth

Implementation Steps

Step 1: Set Up the Kubernetes Cluster

Start with a production-ready EKS cluster using Terraform:

module "eks" { source = "terraform-aws-modules/eks/aws" version = "~> 19.0" cluster_name = "platform-cluster" cluster_version = "1.28" vpc_id = module.vpc.vpc_id subnet_ids = module.vpc.private_subnets eks_managed_node_groups = { platform = { instance_types = ["m5.xlarge"] min_size = 3 max_size = 10 desired_size = 5 } } # Enable IRSA for pod-level IAM enable_irsa = true }

Step 3: Create Application Templates

Provide developers with standardized templates. Here's an example Helm chart structure that encapsulates best practices:

# templates/deployment.yaml apiVersion: apps/v1 kind: Deployment metadata: name: {{ .Values.service.name }} labels: app: {{ .Values.service.name }} team: {{ .Values.team }} spec: replicas: {{ .Values.replicas | default 3 }} selector: matchLabels: app: {{ .Values.service.name }} template: metadata: labels: app: {{ .Values.service.name }} spec: containers: - name: {{ .Values.service.name }} image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}" ports: - containerPort: {{ .Values.service.port }} resources: requests: memory: "128Mi" cpu: "100m" limits: memory: "512Mi" cpu: "500m" livenessProbe: httpGet: path: /health port: {{ .Values.service.port }} readinessProbe: httpGet: path: /ready port: {{ .Values.service.port }}

Key Takeaways

  1. Start small: Begin with GitOps and templates, then add more capabilities
  2. Focus on developer experience: If it's hard to use, developers won't adopt it
  3. Automate guardrails: Use policies and defaults to prevent misconfigurations
  4. Iterate based on feedback: Your platform is a product ‚Äî treat it like one
